# 旅館DX診断アプリ — 要件定義書

## 1. プロダクトコンセプト

### コアアイデア
旅館名を入力するだけで、Google Maps / 楽天トラベル等の**実際の口コミからその旅館固有の「魅力ベクトル」を AI が抽出**し、以降のヒアリング・損失試算・改善策提案すべてを**「魅力を守る × DX で効率化する」の2軸**で評価する。

### 既存DX診断ツールとの差別化
| 観点 | 既存ツール（コンサル / SaaS） | 本アプリ |
| --- | --- | --- |
| 入力 | 業種テンプレに沿った定型質問 | 旅館名 → 実口コミから魅力を自動抽出 |
| 出力 | どの旅館でも同じ改善策 | **その旅館の魅力に応じて結果が変わる** |
| 評価軸 | コスト削減・効率のみ | **「魅力影響度 × DX効果」の2軸** |
| AIの役割 | 文章生成のラッパ | 魅力抽出 → コンテキスト注入 → 構造化評価のフルパス |

### ターゲットユーザ
- 現場スタッフ（無駄に気づいているが言語化できない）
- 経営者 / 若旦那（DX に踏み出したいが「らしさ」を壊したくない）

---

## 2. ユーザーフロー

```
[旅館名入力] 
  → [魅力抽出 API: 口コミ取得 → LLM要約] (3-5秒、ローディング演出)
  → [魅力カード提示: 「あなたの旅館の魅力はこの3つです」確認]
  → [チャット: 抽出した魅力を踏まえて深掘りヒアリング]
  → [診断結果ボタン]
  → [レポート: 損失サマリ + DXマップ + 改善策（魅力影響度付き）]
```

## 3. 機能要件

### 3.1 魅力抽出（新規・差別化のコア）

#### 入力
- 旅館名（必須）
- 所在地（任意、同名旅館の曖昧性解消用）

#### データソース（優先順位順）
1. **Google Maps Places API**（公式・有料）
   - Place Details で口コミ最大5件取得
   - $17 / 1000 リクエスト程度
   - Worker secret に `PLACES_API_KEY`
2. **楽天トラベル API**（公式・無料）
   - 施設検索 + レビュー検索
   - 楽天会員 ID 登録のみで利用可
3. **手動貼り付けフォールバック**
   - 口コミが取得できない / 少ない場合
   - テキストエリアに自由貼り付け

> ⚠ じゃらん・一休は公式APIがなくスクレイピングは規約違反のため使わない。

#### 出力（魅力ベクトル）
```ts
type Charm = {
  facility_name: string;
  location?: string;
  charm_tags: string[];          // 3-5個、例: ["夜の静寂", "女将のもてなし", "源泉かけ流し"]
  charm_summary: string;         // 120字程度の要約
  protect_keywords: string[];    // DX で壊してはいけない要素
  source: 'places' | 'rakuten' | 'manual';
  raw_reviews_count: number;
}
```

#### 抽出ロジック
- OpenAI `gpt-4o-mini` + JSON Schema strict
- システムプロンプト: 「旅館の口コミ群から、その旅館固有の魅力を3-5個のタグと短文要約に圧縮せよ。一般的な「綺麗・美味しい」ではなく、固有名詞や具体的な体験を抽出すること」

### 3.2 チャットへの魅力注入

抽出した `Charm` をシステムプロンプトに埋め込み、ヒアリングを**その旅館の魅力に紐付けて深掘り**させる。

例（システムプロンプトの一部）:
```
あなたが診断している旅館は「○○館」です。
お客様が高く評価している魅力は以下です:
- 夜の静寂（深夜帯の館内が静かで非日常感がある）
- 女将のもてなし（チェックイン時の手書き案内）
- 源泉かけ流し（24時間入浴可能）

ヒアリングでは、業務上の無駄を聞き出しつつ、
「その業務を効率化することがこの魅力を損なわないか」も
さりげなく確認してください。
```

### 3.3 改善策の2軸評価（既存ロジック拡張）

`analyze.ts` の各 improvement に対して以下を追加:

```ts
type Improvement = {
  // 既存
  title: string;
  annual_loss_jpy: number;
  difficulty: 'easy' | 'medium' | 'hard';
  // 新規
  dx_effect_score: number;       // 0-100（既存の損失額・難易度から導出）
  charm_impact: 'safe' | 'caution' | 'risk';  // 魅力との相性
  charm_impact_reason: string;   // 「夜の静寂を壊す可能性あり」等
}
```

評価ロジック:
- LLM に `charm_tags` と改善策タイトル・内容を渡し、矛盾度を3段階で判定
- バッチで一括判定（呼び出し回数削減）

### 3.4 レポート表示

#### 散布図（DXマップ）
- 横軸: DX効果スコア（年間削減額ベース）
- 縦軸: 魅力影響度（safe=上 / caution=中 / risk=下）
- 右上（高効果 × 魅力に安全）= 最優先

#### 改善策カード
- 既存のカードに「魅力影響」バッジを追加（緑/黄/赤）
- バッジクリックで `charm_impact_reason` を表示

## 4. API 契約（追加・変更）

### POST /api/charm（新規）
```
Request:  { facility_name: string, location?: string, manual_reviews?: string }
Response: Charm (上記の型)
Error:    404 (口コミなし) → フロントが手動入力UIに切替
```

### POST /api/chat（変更）
```
Request:  { messages: [...], charm?: Charm }
            charm が渡されたらシステムプロンプトに注入
Response: 既存通り text/plain stream
```

### POST /api/analyze（変更）
```
Request:  { messages: [...], charm?: Charm }
Response: AnalyzeResponse に improvements[i].charm_impact / charm_impact_reason を追加
```

## 7. 非機能要件 / 制約

- Places API のクォータ: 1日100リクエストで初期は十分。Workers の rate limit と組み合わせて保護。
- 口コミ取得失敗時は必ず手動入力フォールバックがある（差別化機能が完全に死なないため）。
- 魅力抽出結果はキャッシュ可能（同一旅館で日次更新で十分）。
